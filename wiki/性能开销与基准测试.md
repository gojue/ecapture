# 性能开销与基准测试

## 概述

eCapture 基于 eBPF uprobe 机制拦截用户空间函数调用，性能开销由以下四部分构成：

1. **Uprobe 进出开销**：内核为每个拦截的函数调用所增加的上下文切换（约 1-2μs/事件）
2. **eBPF 程序执行**：内核态数据提取逻辑（约 0.1-0.5μs/事件）
3. **Perf Buffer 传输**：内核态到用户态的数据拷贝
4. **用户态处理**：事件解析、格式化和写入输出

## 预期性能特征

| 流量场景 | 预期开销 | 说明 |
|---------|---------|------|
| 低流量（< 100 req/s） | 可忽略（< 1% CPU） | uprobe 开销被摊薄 |
| 中流量（100-1K req/s） | 低（1-3% CPU） | perf buffer 容量充足 |
| 高流量（1K-10K req/s） | 中等（3-8% CPU） | 可能需要调整 perf buffer 大小 |
| 极高流量（> 10K req/s） | 显著（> 10% CPU） | 有 perf buffer 溢出风险，建议用 `--pid` 限制范围 |

## 基准测试方法

### 测试环境准备

```bash
# 记录系统信息
uname -a
cat /proc/cpuinfo | head -20
free -h
openssl version
ls -la /sys/kernel/btf/vmlinux
```

### 使用 wrk 进行基准测试

#### 安装 wrk

```bash
sudo apt-get install wrk
```

#### 基线测试（无 eCapture）

```bash
wrk -t4 -c100 -d60s https://localhost:8443/
```

#### Text 模式测试

```bash
# 终端 1：启动 eCapture
sudo ecapture tls -m text --pid=<server_pid>

# 终端 2：运行基准测试
wrk -t4 -c100 -d60s https://localhost:8443/
```

#### PcapNG 模式测试

```bash
# 终端 1
sudo ecapture tls -m pcap -i lo --pcapfile=/tmp/bench.pcapng --pid=<server_pid>

# 终端 2
wrk -t4 -c100 -d60s https://localhost:8443/
```

### 关键指标

| 指标 | 测量方法 | 工具 |
|------|---------|------|
| CPU 开销（目标进程） | 对比启用/未启用 eCapture 时的 CPU 使用率 | `pidstat -p <pid> 1` |
| eCapture CPU 使用 | eCapture 进程的 CPU 消耗 | `pidstat -p <ecapture_pid> 1` |
| 内存使用 | eCapture 进程的 RSS | `ps -o rss= -p <ecapture_pid>` |
| 延迟影响 | p50/p99 延迟增加 | `wrk --latency` |
| 吞吐影响 | Requests/sec 下降比例 | `wrk` 输出 |
| 事件丢失率 | perf buffer 溢出事件 | eCapture 日志（"lost events"） |

## Perf Buffer 调优

eCapture 默认 perf buffer 大小为 **4 MB/CPU**。高流量场景下可能出现 "lost X events" 消息，表示用户态读取跟不上内核态写入。

### 缓解策略

1. **限制范围**：使用 `--pid` 仅捕获特定进程
2. **选择轻量模式**：`keylog` 模式的数据量远小于 `text` 模式
3. **增大 perf buffer**：通过 `--per_cpu_map_size` 调整（如支持）

## 自动化基准测试脚本

```bash
#!/bin/bash
# benchmark.sh — eCapture 性能基准测试
set -euo pipefail

if [[ "$(uname -s)" != "Linux" ]]; then
    echo "ERROR: 本脚本只能在 Linux 上运行" >&2
    exit 1
fi

TARGET_URL="${1:?用法: $0 <https_url> [pid]}"
TARGET_PID="${2:-}"
DURATION="60s"

echo "=== eCapture 性能基准测试 ==="
echo "时间: $(date -Iseconds)"
echo "内核: $(uname -r)"
echo "目标: ${TARGET_URL}"

# 基线
echo "--- 基线（无 eCapture）---"
wrk -t4 -c100 -d${DURATION} --latency "${TARGET_URL}" | tee /tmp/bench_baseline.txt

# eCapture text 模式
echo "--- eCapture（text 模式）---"
PID_FLAG=""
[[ -n "${TARGET_PID}" ]] && PID_FLAG="--pid=${TARGET_PID}"
sudo ecapture tls -m text ${PID_FLAG} &>/tmp/ecapture_bench.log &
ECAP_PID=$!
sleep 3

wrk -t4 -c100 -d${DURATION} --latency "${TARGET_URL}" | tee /tmp/bench_ecapture.txt
sudo kill ${ECAP_PID} 2>/dev/null || true

echo "=== 结果已保存到 /tmp/bench_baseline.txt 和 /tmp/bench_ecapture.txt ==="
```

## 已知限制

- uprobe 仅在函数入口和返回处触发，不影响函数内部执行
- perf buffer 使用内核内存，大量并发可能增加内核内存压力
- CO-RE 模式相比非 CO-RE 模式在加载阶段略慢（需要 BTF 解析），但运行时无差异
